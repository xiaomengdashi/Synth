# 关于《使用大模型总结 B 站视频》方案的可行性分析与实施计划

## 1. 方案可行性分析

您提供的文章 [使用大模型总结B站视频](https://www.imrising.cn/posts/2024/0913) 介绍了一种基于 LangChain 的 `BiliBiliLoader` 和 RAG（检索增强生成）技术的 B 站视频总结方案。

针对我们的 `Synth` 项目，该方案的可行性和优缺点分析如下：

### 1.1 可行且极具参考价值的点（建议采纳）
1. **引入身份认证 (Cookies) 绕过风控与获取更多字幕**：
   - **现状**：我们目前的系统是“匿名”抓取 B 站字幕的。但 B 站对于部分视频的 AI 生成字幕，或者部分高频请求，会要求必须登录。
   - **文章方案**：文章指出需要获取 `SESSDATA`、`bili_jct`、`buvid3` 这三个核心 Cookie 参数。
   - **结论**：**极度可行且必要**。在我们的请求头中加入这三个 Cookie 参数，可以大幅提高字幕提取的成功率，并能获取到仅限登录用户可见的 AI 字幕。

### 1.2 可行但建议优化的点（需根据实际情况调整）
1. **使用 LangChain 的 `BiliBiliLoader`**：
   - **文章方案**：依赖 `langchain_community` 和底层的 `bilibili-api-python` 库。
   - **结论**：**可行，但不建议全盘照搬**。引入 LangChain 会导致后端依赖变得极其沉重。我们刚才已经在 `tasks.py` 中用非常轻量级的 `httpx` 实现了等效的官方 API 调用（获取 cid -> 获取 subtitle url -> 下载拼接）。我们只需要在现有的轻量级代码上补充 Cookie 即可，没必要为了一个 Loader 引入庞大的 LangChain 框架。
2. **基于 TextSplitter 和 RAG 的总结方式**：
   - **文章方案**：将长字幕切分成小块（Chunk），存入向量数据库，然后使用 RAG（检索增强）来提问。
   - **结论**：**部分可行，但 RAG 不适合“全文总结”**。RAG 的原理是“相似度检索”，如果让它“总结全文”，它只会检索出几个片段，导致总结丢失全局视野。考虑到目前的主流模型（如 GPT-4o, DeepSeek, Claude 3.5）都支持 128K 甚至更长的上下文，直接将完整的字幕文本输入给大模型效果最好。对于超长视频，采用 Map-Reduce（分段总结再汇总）比 RAG 更合适。

## 2. 拟定实施方案

基于上述分析，为了吸收该文章的精髓并保持我们 `Synth` 项目的轻量与高效，我计划进行以下改造：

### 步骤一：增加 B 站 Cookie 身份认证支持
1. 修改 `backend/app/routers/tasks.py` 中的 B 站 API 请求逻辑。
2. 在 `httpx.AsyncClient` 的请求中，支持读取并挂载 `SESSDATA`、`bili_jct`、`buvid3` 等 Cookie 信息（可以通过环境变量或直接在代码/配置中预留位置）。
3. 这样可以确保我们的爬虫伪装成真实登录用户，从而获取到完整的 AI 专属字幕。

### 步骤二：完善长文本的截断与安全处理
1. 吸收文章中对“长文本边界”的考虑。目前的提示词中我们粗暴地截断了 `[:20000]` 个字符。
2. 我们将引入一个基于 Token 估算或按段落/标点符号进行安全截断的逻辑，确保丢给 LLM 的文本既包含足够的信息，又不会因为被硬生生切断而在结尾出现乱码。

### 步骤三：不引入 LangChain，保持架构极简
1. 继续沿用当前原生 `httpx` + 原生 `OpenAI SDK` 的架构。
2. 保持系统的高性能与低延迟，避免向量数据库（Chroma）带来的额外内存开销和部署复杂性。
